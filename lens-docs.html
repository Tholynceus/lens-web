Getting started

Overview

LENS turns any X profile or Base contract into a plain-language rug-risk verdict by reading the deployer wallet on-chain, instead of trusting whatever the bio says. Bios lie, chains don't.

a bio claims one thing, the deployer wallet tells the truth

The problem it solves

Every day on crypto X, a profile posts a contract address, promises a locked LP and a long-term build, and asks you to ape. The bio is free to write and easy to fake. The only thing that cannot be faked is what the deployer wallet has actually done on-chain, how many tokens it has launched, whether it sold them, where it got its funding, and who it is quietly linked to.

Normally, checking that means copying the address into a block explorer, clicking through transactions, cross-referencing wallets, and piecing the story together yourself, for every single token. Most people do not have the time, so they trust the bio and get rugged.

What LENS does

LENS does that investigation for you, automatically, the moment you open a profile. It reads the bio for a wallet, a contract or a GitHub link, resolves the person behind it, pulls their real on-chain history, and collapses everything into one of three calls you can act on in a second.

CLEAR  nothing alarming on-chain CAUTION  mixed signals, look closer STOP  behavior that rugs people

LENS reads Base chain and tokens launched through Bankrbot. Multi-chain support is on the roadmap.

Two ways to use it

Most people use the Chrome extension, which runs on x.com automatically and shows the verdict inline under any profile. Builders and agents use the public API and MCP server to run the exact same check from a script, a bot, or mid-conversation in an AI agent.

Getting started

Install the extension

LENS is a Chrome MV3 extension. Install it once and it runs on x.com automatically, with nothing to configure and no account to create.

Install in three steps

  1. Add to Chrome. Install from the Chrome Web Store, or load the unpacked build from the repo if you want the very latest version before it ships to the store.
  2. Open any X profile. LENS detects the page, reads the bio, and injects a panel directly under it within about a second. There is no button to press.
  3. Read the verdict. The trust score, the verdict chip, and the dev's deploy history all appear inline, exactly where you are already looking.

Permissions, in plain terms

LENS asks for the minimum it needs to do its job, and nothing that could touch your funds. Here is what each permission is actually for.

PermissionWhat it is actually for
x.com accessRead the bio on the profile you are viewing and inject the panel under it
storageRemember your settings and cache recent scans so the same profile loads instantly
networkCall the LENS API to fetch on-chain data for the address it found

LENS is read-only. It never asks to connect a wallet, never holds private keys, and never signs a transaction. If anything ever prompts you to connect a wallet in the name of LENS, it is not LENS.

Staying updated

The extension updates itself through the store. New intelligence features are added on the backend, so most improvements reach you without any update at all, the panel simply starts showing more.

Getting started

Your first scan

Open the profile of any token dev on Base and the panel appears under the bio with everything LENS could read about the deployer. Here is how to read it.

the panel LENS injects under a profile

Anatomy of the panel

The panel is built to be scanned in a glance, with the headline call on the left and the supporting evidence to its right.

  • Trust score — a 0 to 100 read on the deployer. Higher is safer. It starts high and drops for each red flag in the wallet's history.
  • Verdict chip — the headline CLEAR, CAUTION or STOP. This is the one thing to read if you only read one thing.
  • Stat row — the key numbers: how many tokens the dev has deployed, how many they sold, any linked accounts, and where the wallet was funded from.
  • Red lines — the specific behaviors that pulled the verdict down, named in plain language so you know exactly why.

When a profile is empty

If a bio has no wallet, no contract and no GitHub, there is nothing on-chain to read, and LENS says so plainly rather than inventing a score. An empty profile is information too: a token dev with zero on-chain footprint is unproven, not safe.

Treat "not found" as "not yet proven", and weigh it like any other CAUTION signal.

How verdicts work

The verdict system

Every scan collapses to one of three calls. The logic is deliberately simple so it cannot be gamed, and so you can trust it at a glance.

the worst signal sets the call

Worst signal wins

LENS does not average its signals. A single STOP-level red line is enough to mark the whole token STOP, even if nine other checks are green. This is on purpose: a token where the dev can freeze your balance is dangerous no matter how clean the rest of the picture looks, and an averaged score would hide that one fatal flaw behind a comfortable-looking number.

The three calls

VerdictMeaningWhat to do
CLEARNo red lines tripped on-chainStill your call, but nothing here screams rug
CAUTIONSoft signals worth a closer lookRead the red lines before you size in
STOPBehavior that has rugged holdersWalk away, or treat it as a pure gamble

Why three and not a number

A raw score of 63 out of 100 tells you very little in the heat of a launch. CLEAR, CAUTION and STOP map to a decision you can actually make in the two seconds you have before a token moves. The numeric trust score is still there underneath for when you want the detail, but the verdict is what you act on.

How verdicts work

The seven red lines

For a token, LENS runs a transparent rule set called tokenRedLines(). Each rule maps to a flag, and the highest flag becomes the verdict. Nothing is a hidden black-box score, you can always see exactly which line tripped.

The seven lines

Red lineFlagWhy it matters
Dev sold a prior tokenSTOPA deployer who dumped on holders before will very likely do it again
Serial deployerSTOPDozens of launches in a short window is a rug-factory pattern, not a builder
Dev claimed fees and leftSTOPTook the creator fees with no follow-through, the soft rug
Bundled supplyCAUTIONLaunch wallets quietly hold a large share, so a coordinated dump can hit any time
Fresh funding trailCAUTIONWallet was funded minutes before launch from a fresh CEX withdrawal, a throwaway
Hidden linked accountsCAUTIONAlt identities tied to past failed launches that the dev is not advertising
Thin or no historyCAUTIONNothing on-chain to verify, so the dev is unproven and should be treated as such

How a verdict is built

LENS evaluates all seven lines on every token, collects the flags, and takes the worst one. If any STOP line trips, the token is STOP. If no STOP trips but at least one CAUTION does, it is CAUTION. If nothing trips, it is CLEAR. Because the rules are explicit, you are never asked to trust a mystery number, you can read the exact line and decide whether you agree.

A CLEAR verdict means no red line tripped, not a guarantee of safety. New scams invent new patterns. Always do your own research before aping.

How verdicts work

Trust score

For a dev or a wallet, LENS also returns a 0 to 100 trust score, a quick numeric read on the person behind the token, built from the same on-chain signals as the red lines.

How it is built

The score starts high and is pulled down by each flag in the wallet's history. A long record of honest launches where the dev held through keeps the score up. A pattern of dumps, claimed fees, fresh funding and dead tokens drags it down fast. It is a summary of behavior over time, not a snapshot of one launch.

Reading the bands

RangeBandRead
80 – 100solidClean, consistent history, holds through launches
45 – 79mixedSome flags present, read the detail before trusting
0 – 44riskyStrong rug-pattern history, treat with extreme care

Score versus verdict

The verdict is the call to act on; the score is the nuance behind it. A dev can sit at a respectable 70 and still earn a STOP if one specific fatal line trips, because worst-signal-wins applies to the verdict regardless of the average. Use the score to compare two devs who both come back CAUTION, and use the verdict to decide whether to touch either of them at all.

Scanning

Scan an X profile

This is the default mode and the reason most people install LENS. Open a profile and the verdict appears, you do nothing.

How it works

When you land on an x.com profile, LENS reads the bio for the things that map to an identity on-chain: a wallet address, a token contract, or a GitHub handle. It resolves whichever it finds to the deployer behind it, runs the full set of checks, and injects the verdict panel under the bio, all in about a second.

Because the read is automatic, you get the same scrutiny on every profile without ever deciding to investigate. The check you would normally skip happens by default.

Case-insensitive matching

Addresses get written into bios in all sorts of formats, fully checksummed, all lowercase, or somewhere in between. LENS matches them case-insensitively, so a checksummed address and its lowercase form resolve to the exact same dev. This sounds small, but a single mismatched character used to be the difference between a correct verdict and a false "not found".

You can also scan a handle directly through the API with ?username=, which resolves the bio to a wallet first, then runs the same checks the extension does.

Scanning

Scan a contract

You can point LENS straight at a token contract, in the extension or through the public API, and get the same verdict and red lines for whoever deployed it.

From the command line

bash
# scan a Base token contract
curl "https://lens-liard.vercel.app/api/lookup?contract=0x9a3c…b7e2"

From JavaScript

javascript
const res = await fetch(
  `https://lens-liard.vercel.app/api/lookup?contract=${ca}`
)
const { data } = await res.json()

if (data.found) console.log(data.verdict, data.red_lines)
else console.log('no on-chain record yet')

Scanning

Scan a wallet

Skip the token and read the dev directly. Pass a wallet address to get its full deploy history, sell behavior, funding trail and trust score, the whole picture behind the person.

Reading a wallet

bash
# read a deployer wallet directly
curl "https://lens-liard.vercel.app/api/lookup?wallet=0x8618…C2BE"

When to use it

Scanning a wallet is the move when you already have an address and want the dev's track record rather than one token's verdict. It answers the questions that matter most before a launch: how many tokens has this wallet shipped, how did each one end, did the dev sell, and where did the gas come from.

It is also how you connect the dots between identities. Two different X profiles that resolve to the same wallet are the same operator, no matter how the bios are written.

All three inputs, ?username=, ?contract= and ?wallet=, return the same response shape, so you can build one handler for all of them.

Scanning

B20 tokens

B20 is Base's native token standard. Unlike a normal ERC-20, it bakes issuer powers straight into the token, so LENS reads a different and more dangerous set of signals.

a B20 scan reads issuer powers, not just balances

Why B20 needs its own scanner

A normal ERC-20 is just a deployed contract, and the risk lives in the dev's behavior. A B20 is a precompile with built-in issuer powers: the issuer can freeze and seize balances, mint without limit, pause transfers, and gate who is allowed to send or receive. These powers are part of the token itself, so a token can look perfectly fine on the surface and still let its issuer freeze your bag at will.

A deployer-history scan does not see any of this. LENS reads the powers directly off the precompile and turns them into the same CLEAR, CAUTION, STOP language.

What LENS reads

The scanner asks the B20 factory whether a contract is a B20, then reads its state directly, read-only, no key, no signature. Transfer-gating plus a seizable asset is a STOP. A paused transfer is a STOP. An uncapped supply is a CAUTION. Restricted minting is informational.

Intelligence

Dev claim & fees

On Bankrbot launches the creator can claim the trading fees a token accrues. Whether and how a dev claims is one of the clearest tells of intent.

How it works

As a token trades, fees build up that the deployer is entitled to claim. LENS reads whether the dev has claimed, roughly how much, and crucially what they did afterward. A dev who claims a healthy fee balance and keeps shipping is behaving like a builder. A dev who claims and immediately goes silent has taken their money and left.

Why it is a STOP

Claiming fees and disappearing is the textbook soft rug: there is no dramatic liquidity pull to point at, the token just quietly dies while the dev moves on to the next one. Because it is so reliable a signal, that pattern alone trips a STOP red line, and it usually shows up next to a serial-deployer flag, because the same wallet tends to do it again and again.

Intelligence

Serial deployer

LENS counts every token a wallet has ever deployed and how each one ended. The shape of that history separates a real builder from a rug factory.

What counts as serial

Shipping a few tokens over time is normal, plenty of honest devs iterate. The signal is volume plus outcome: dozens of tokens launched in a short window, most of them dead or dumped, is not iteration, it is a production line for exit liquidity. LENS looks at both the raw deploy count and the fate of each token, not just the number.

How it feeds the verdict

A serial-deployer pattern trips a STOP and weighs heavily on the trust score. It is one of the strongest predictors LENS has, because past behavior across many launches is far harder to fake than any single token's chart.

Intelligence

PleaseBro detection

"PleaseBro" tokens come from a known farm pattern that mass-produces near-identical copy tokens to bait buyers. LENS flags wallets and contracts that match the fingerprint.

The pattern

These launches share a recognizable signature in how they are deployed and funded, designed to look like a fresh organic token while being one of hundreds churned out by the same operation. The whole point is volume: spin up enough lookalikes and some will catch a bid before anyone notices the dev has done this five hundred times.

What LENS does with it

When a deployer or contract shows the PleaseBro signature, the verdict reflects it before you have read a single reply in the thread. It is the kind of pattern that is invisible if you are only looking at one token, and obvious once you can see the wallet's full footprint, which is exactly what LENS gives you.

Intelligence

CA & bio history

Devs swap the contract address in their bio between launches. LENS keeps a history of which CAs a profile has promoted over time, so a "first ever token" claim is easy to disprove.

Why bios are not enough

A bio shows you the current contract, never the last five. A profile can promote a token, let it die, edit the bio, and promote the next one as if it were the first, and to anyone seeing it fresh, it is. LENS tracks the addresses a profile has carried over time, so the pattern of serial relaunches becomes visible instead of being quietly erased with each bio edit.

Reading the history

A profile that has cycled through several contracts in a month is its own warning, regardless of what any single token's chart looks like. The history turns a "new project" into "the same person's sixth attempt", which is usually all you need to know.

Intelligence

Bundled wallets

At launch, insiders often spread supply across many wallets in a single block to hide how much they really control. LENS detects bundled buys and estimates the true insider share.

The trick

If one wallet bought 30% of supply at launch it would be obvious. So instead, supply is split across a bundle of wallets that all buy in the same block, each holding a slice small enough to look like an ordinary buyer. Together they still control a huge share, and they can dump it in a coordinated move whenever they choose.

What LENS shows

LENS detects the bundle and estimates how much of the supply the launch wallets actually hold between them. A high bundled share is a CAUTION line, because the float you think is in public hands may be sitting in wallets ready to exit all at once.

Intelligence

AI verdict

On top of the rule-based red lines, LENS can run an AI pass that reads the full on-chain picture and writes a short, plain-language summary of why a dev looks safe or sketchy.

Rules decide, AI explains

The verdict itself always comes from the transparent rule set, so it stays predictable and cannot be talked around. The AI layer sits on top to do something the rules cannot: read the whole context at once and explain it in a sentence or two you can paste straight into a group chat, "deployed seven tokens, sold two within hours, funded from a fresh wallet, treat with care".

Why keep them separate

Keeping the decision in the rules and the explanation in the AI means you never get a verdict you cannot trace back to a concrete on-chain fact. The AI makes the result readable; it does not get a vote on whether something is a STOP.

Intelligence

Token health

Beyond the dev, LENS reads the token itself, its liquidity, holder spread, and whether the LP is locked or sitting somewhere it can be pulled.

The dev and the token are two questions

A trustworthy dev can still launch a token with unlocked liquidity, and a sketchy dev can launch one that is technically fine. LENS keeps the two readings separate so you see both: the deployer's history on one side, the token's own health on the other.

What it checks

Liquidity depth tells you how easily the token can be sold without collapsing the price. Holder spread shows whether the supply is genuinely distributed or concentrated in a few hands. LP lock status answers the most basic rug question of all, can the liquidity simply be removed. A great dev on a token with unlocked LP is still a risk, and LENS will not let one hide the other.

Intelligence

Linked accounts

Devs run alt accounts to recycle a reputation after a failed launch. LENS surfaces the X accounts tied to the same wallet or funding source, so a "new" builder with a buried history is exposed.

How identities get linked

People are easy to rebrand; wallets and funding trails are not. When a dev abandons a burned account and starts fresh, the new profile still tends to touch the same wallets, or get funded from the same source. LENS follows those on-chain threads to connect a clean-looking new account back to the launches it would rather you did not see.

Why it matters

The whole value of a reputation is that it cannot be reset on a whim. Linked-account detection restores that: a builder with a real track record keeps it, and a serial rugger cannot simply open a new handle and start the con over with a blank slate.

Intelligence

Funding trail

LENS traces where the deployer wallet got its gas. A wallet funded minutes before launch from a fresh exchange withdrawal is a throwaway, very different from one with months of real history.

Following the money in

Every wallet had to be funded by something. LENS looks at where the deployer's first funds came from and when. A wallet that received a single CEX withdrawal moments before deploying a token has no history to stand on, it was created for this launch and will likely be discarded after it.

The signal

A fresh funding trail is a CAUTION line on its own, and a strong one when it lines up with bundled supply or a serial-deployer pattern. A wallet with a long, varied funding history is the opposite signal: someone who has been operating on-chain for a while and has more to lose than one rugged launch.

Intelligence

Smart followers

LENS labels the on-chain-relevant accounts that follow a profile, so you can tell organic traction from a padded follower count.

Followers are not all equal

A follower count is trivially inflated and tells you almost nothing. What matters is who is actually paying attention: real traders, known deployers, and recognizable names. LENS reads the meaningful followers and gives each a short role label generated from its own on-chain footprint, so a profile's real standing in the ecosystem becomes visible underneath the raw number.

What it tells you

A new token followed by a handful of serious traders is a very different signal from one followed by ten thousand empty accounts. Smart followers let you read the quality of the audience, not just its size.

Intelligence

GitHub intel

If a bio links a GitHub, LENS reads the public profile, the repos, the account age and the activity, and shows whether the "open source" claim holds up.

Checking the code claim

"Fully open source, audited, doxxed dev" is one of the most common lines in a bio, and one of the easiest to fake with an empty account created last week. LENS reads the linked GitHub the same way it reads a wallet: how old is the account, what is actually in the repos, and is there real, sustained activity, or one empty placeholder pushed the day before launch.

Why it belongs next to the on-chain read

Code credibility and on-chain behavior together give a fuller picture of a builder than either alone. A dev with a real, active GitHub and a clean deploy history is a genuinely different prospect from an anonymous wallet with a freshly minted repo.

Intelligence

Community tags

Holders can tag wallets and profiles, for example scammer, builder, or known team. Tags are community-voted, so a single bad actor cannot brigade a label.

Crowd memory

On-chain data tells you what a wallet did; community tags tell you what people learned the hard way. A wallet that rugged a community last month can be tagged so the next person sees the warning before they engage, and a genuine team can be tagged as known so they are not mistaken for an impersonator.

Why voting matters

Tags are weighted by community votes rather than set by any single person, which makes them hard to abuse in either direction, you cannot smear an honest builder with one malicious tag, and you cannot whitewash a scammer with one fake endorsement. The tags appear alongside the on-chain verdict, so social knowledge and chain data sit side by side.

MCP & agents

LENS ships an MCP server so any agent can run a rug check mid-conversation, the same verdict you see, available as a tool call.

Connect it

bash
claude mcp add --transport http lens https://lens-liard.vercel.app/api/mcp
an agent calls LENS like any other tool

The three tools

The server exposes three tools, scan an X username, scan a contract, and scan a wallet. Each returns the verdict and red lines as structured data, so an agent can read the result and reason about it rather than just printing a string.

This is what lets an agent vet a token in the middle of a task: it sees a contract address, calls the scan tool, gets back a STOP with the reasons, and can refuse to ape on your behalf, all without you switching tabs.

The full agent guide lives on the Agents page.

For developers

Response schema

Every lookup returns the same shape, whether you asked by username, contract or wallet. Fields that do not apply are omitted rather than returned as null.

Fields

FieldTypeMeaning
successboolThe request resolved
data.foundboolAn on-chain footprint exists
data.verdictstringCLEAR · CAUTION · STOP
data.trust_scorenumber0–100 (wallet and dev reads)
data.red_linesarrayEach is { flag, label, detail }
data.deployer_walletstringThe resolved deployer address
data.token_symbolstringToken ticker, when scanning a contract
data.sourcestring"b20" on B20 reads

Handling the response

Check success first, then data.found. If found is false, there is simply no record yet, treat it as unproven rather than safe. If found is true, verdict is the headline and red_lines is the evidence. Because the shape never changes across input types, you can write one parser and reuse it everywhere.

Architecture

LENS is a thin client over a small, honest backend. The extension and agents call one API; the API reads the chain and a launch index, with an optional AI pass.

one API, reading the chain and a launch index

How a scan flows

The extension or an agent sends an address to the LENS API. The API reads on-chain history through Alchemy on Base, checks a launch index in Supabase for tokens shipped through Bankrbot, and optionally runs the picture through Venice for a plain-language AI summary. The red-line rules run over all of it, and the verdict comes back.

The pieces

  • Alchemy provides the Base RPC, the raw on-chain reads, deploys, transfers, funding.
  • Supabase holds the bankr_launches index, so a token can be matched to its launch without crawling the whole chain.
  • Venice powers the optional AI verdict that explains the result in a sentence.
  • lens-listener is a background worker that polls for new launches and keeps the index current, so fresh tokens are scannable quickly.

Everything the API touches is public data. There is no wallet connection anywhere in the flow, and nothing LENS reads requires a signature.

FAQ

Short answers to the questions people ask most.

Does LENS touch my wallet?

No. LENS is strictly read-only. It never connects a wallet, never holds keys, and never signs a transaction. If anything ever asks you to connect a wallet in the name of LENS, it is an impersonator.

Which chain does it cover?

Base, focused on tokens launched through Bankrbot, plus B20 reads once that standard activates. More chains are on the roadmap.

A token shows "not found", is that bad?

It means LENS has no on-chain record for it yet, which is common for brand-new or non-indexed tokens. Treat unknown as unproven, not as safe.

Is a CLEAR verdict a guarantee?

No. CLEAR means no red line tripped, not that the token is safe. New scams invent new patterns, so always do your own research before aping.

How fresh is the data?

On-chain reads are live. The launch index is kept current by a background listener, so new tokens become scannable shortly after they launch.
